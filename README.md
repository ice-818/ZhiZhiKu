# 智知库 (ZhiZhiKu)

**互联网企业私有知识库智能问答系统 —— 让 PRD、接口文档、运维手册「问一句，就有据可查」**

智知库是一款面向**互联网企业内部场景**的私有化 RAG 问答系统。产品迭代快、团队协作密、文档体量大，但员工仍要在 Confluence、飞书、GitLab、工单系统之间来回翻找。智知库将 PRD、API 文档、On-Call 手册、客服话术等内部资料统一入库，研发、产品、运营、客服用自然语言提问即可获得**有出处、可核对**的回答，全程数据不出企业内网。

---

## 面向互联网企业，我们解决什么问题？

互联网公司文档增长快、版本迭代猛、人员流动大，**知识跟不上业务节奏，公有云 AI 又不敢用**。智知库针对以下互联网行业真实痛点设计：

### 痛点一：文档散在多平台，研发和产品「到处找、总问人」

**典型场景**：PRD 在飞书文档，接口说明在 Swagger / GitLab Wiki，架构图在 Confluence，故障处理在 Notion——新人 onboarding、跨端联调、产品改需求时，不知道最新版在哪。

| 互联网现状 | 带来的问题 | 智知库的解决方式 |
|------------|------------|------------------|
| 知识分散在飞书 / Confluence / Git / 群聊 | 入口不统一，「问人」比「搜文档」更快 | 按业务线或系统建立知识库，文档集中入库 |
| 平台内搜索只能匹配标题和关键词 | 搜不到口语化提问（如「支付回调超时怎么处理」） | 语义检索 + 关键词混合检索，理解研发/运营提问习惯 |
| 一份 PRD / 设计文档几十页 | 改一个小需求也要翻全文找背景 | 自动定位相关段落，回答附带原文引用 |

**解决效果**：减少「@ 老员工」「拉会同步」，让研发、产品、测试自助查资料，缩短协作等待时间。

---

### 痛点二：产品迭代快，文档追不上版本

**典型场景**：双周发版、灰度频繁，接口字段改了文档没更；上一版 PRD 还在被引用；客服按旧话术答复引发客诉。

| 互联网现状 | 带来的问题 | 智知库的解决方式 |
|------------|------------|------------------|
| 发版节奏快，文档更新滞后 | 员工按过期信息执行，引发线上问题 | 按版本 / 迭代维护知识库，更新后重新入库 |
| 同一功能多份 PRD、方案稿并存 | 不知道以哪份为准 | 分库管理（如「支付 v2」「会员 3.0」），避免混用 |
| Release Note 没人看 | 变更信息无法有效触达全员 | 将发版说明入库，自然语言查询「这次改了什么」 |

**解决效果**：让文档跟得上迭代节奏，降低因**信息过期**导致的联调返工和对外口径错误。

---

### 痛点三：客服 / 运营 / 售前重复答疑，人力跟不上业务扩张

**典型场景**：用户问会员权益、活动规则、退款流程，客服翻帮助中心；运营问活动配置规则；BD 问产品能力与报价边界——同一问题每天重复上百次。

| 互联网现状 | 带来的问题 | 智知库的解决方式 |
|------------|------------|------------------|
| 帮助中心检索体验差，客服培训周期长 | 大促期间人力吃紧，响应变慢 | 将 FAQ、活动规则、话术库入库，统一问答入口 |
| 不同客服答复口径不一致 | 引发客诉与品牌风险 | 答案基于官方文档生成，对外口径统一 |
| 新业务线扩张，支持成本线性增长 | 人力 ROI 下降 | 7×24 内部自助问答，客服先查系统再答复用户 |

**解决效果**：把「人背话术、人翻文档」变成「系统答标准题」，释放客服、运营重复劳动，大促期间也能稳住响应质量。

---

### 痛点四：On-Call 和故障处理，关键时刻找不到 Runbook

**典型场景**：凌晨告警，on-call 工程师在多个 Wiki 里找处理步骤；中间件、业务、基础设施文档各管各的；老事故复盘报告沉在群里没人翻。

| 互联网现状 | 带来的问题 | 智知库的解决方式 |
|------------|------------|------------------|
| Runbook 分散，告警时搜索耗时 | MTTR 拉长，故障影响面扩大 | 将 SRE 手册、故障预案、复盘报告入库 |
| 口语化提问搜不到（如「Redis 主从切换失败」） | 依赖值周老手经验，新人难顶上 | 语义检索理解故障现象，快速定位处理步骤 |
| 复盘结论没有沉淀为可检索知识 | 同类故障反复踩坑 | 事故复盘文档入库，支持「类似问题以前怎么解决的」 |

**解决效果**：缩短故障定位与恢复时间，降低 on-call 对**个人经验**的过度依赖。

---

### 痛点五：用户数据与商业机密不能用公有云 AI

**典型场景**：公司明文禁止将 PRD、用户画像方案、核心算法说明上传 ChatGPT；安全团队通不过外部知识库 SaaS 采购；开发内网隔离环境无法访问外网 AI。

| 互联网现状 | 带来的问题 | 智知库的解决方式 |
|------------|------------|------------------|
| 代码、PRD、数据方案涉及商业机密 | 员工想用 AI 提效，但合规红线明确 | **100% 私有化部署**，数据存于企业自有服务器 |
| 用户隐私数据不出域要求 | 公有 SaaS 无法通过安全评审 | 支持内网独立部署，不依赖外网 |
| 开发 / 测试隔离网 | 云端 AI 完全不可用 | 支持本地开源大模型，**内网离线可运行** |

**解决效果**：在**满足互联网安全合规要求**的前提下，让研发、产品、运营也能用上内部 AI 问答。

---

### 痛点六：通用大模型「胡说」，不敢用于业务决策和对外答复

**典型场景**：问接口限流阈值，AI 编造了不存在的配置；问活动规则，AI 给出错误优惠条件——用于客服答复或研发排查会直接引发线上事故。

| 互联网现状 | 带来的问题 | 智知库的解决方式 |
|------------|------------|------------------|
| 通用大模型不了解公司内部实现 | 回答泛泛，解决不了联调、排查问题 | 基于 RAG，**只从企业知识库检索后作答** |
| 大模型幻觉，编造不存在的接口或规则 | 误导研发与客服，存在业务与合规风险 | 回答附带引用片段，可回溯 Wiki / PRD 原文 |
| 每次排查需手动粘贴日志说明、设计文档 | 操作繁琐，无法在日常流程中规模化使用 | 知识库持久化，多轮对话自动关联上下文 |

**解决效果**：AI 回答**有据可查、可人工复核**，适用于接口查询、规则确认、故障排查等对准确性敏感的场景。

---

### 痛点七：人员流动大，业务知识随人走

**典型场景**：核心业务 owner 离职，接手人要从零散文档和聊天记录里「考古」；项目关停后方案无人维护；校招 / 社招新人熟悉业务周期长。

| 互联网现状 | 带来的问题 | 智知库的解决方式 |
|------------|------------|------------------|
| 关键逻辑在老员工脑子里，文档残缺 | 交接成本高，接手后易踩坑 | 推动 PRD、设计、Runbook 结构化入库 |
| Wiki 写了但没人看，利用率低 | 「有文档」≠「新人能上手」 | 自然语言问答，降低文档使用门槛 |
| 业务线调整频繁，历史方案难追溯 | 重复造轮子，决策缺依据 | 项目文档、复盘、ADR 可检索、可传承 |

**解决效果**：缓解互联网行业高流动率下的**知识断层**，缩短新人熟悉业务与系统的周期。

## 项目目的

智知库在互联网企业内部的核心目标：

1. **让研发、产品、运营「问文档」而不是「翻 Wiki、@ 人」**  
   用自然语言提问，秒级获取答案，并能看到引用出处。

2. **让知识跟得上快速迭代**  
   按版本、业务线维护知识库，减少过期文档带来的联调返工和对外口径错误。

3. **在合规前提下推动内网智能化**  
   用户数据与商业机密不出域，满足互联网安全团队对私有化部署的要求。

4. **降低高流动率下的知识传承成本**  
   把散落在飞书、Confluence、Git 里的经验变成可检索、可传承的组织资产。

5. **以可快速试点的方式落地，避免长周期自研**  
   为技术团队提供可一周内完成 PoC 的内部知识问答基础平台。

## 价值总结（给技术负责人 / 业务负责人）

| 维度 | 使用前 | 使用智知库后 |
|------|--------|--------------|
| **协作效率** | 多平台翻文档、群里 @ 人、等回复 | 统一入口，自然语言即问即答 |
| **迭代风险** | 文档过期导致联调返工、客服口径错误 | 按版本维护知识库，信息与时效对齐 |
| **故障响应** | On-Call 翻多个 Wiki，依赖老手经验 | Runbook 语义检索，缩短 MTTR |
| **支持成本** | 客服 / 运营重复答同类问题 | 内部先查系统，对外口径统一 |
| **数据安全** | 公有云 AI 触碰用户数据与商业机密红线 | 内网私有化，自主可控 |
| **新人上手** | 熟悉业务与系统需数周 | 自助查询 PRD / 接口 / 规范，缩短 onboarding |

## 核心功能

| 模块 | 互联网场景下的作用 |
|------|-------------------|
| **知识库管理** | 按业务线 / 系统 / 版本分库（如「交易」「增长」「基础架构」） |
| **RAG 对话** | 基于指定知识库问答，返回引用来源，便于研发 / 客服核对 |
| **多功能对话** | 通用对话与 Agent 工具，辅助日常分析与办公 |
| **混合检索** | 提升 PRD、技术文档、中文 Runbook 的召回准确率 |
| **多模型切换** | 内网本地模型或公司已接入的 API 均可配置 |
| **MCP 扩展** | 按需对接内部工单、监控、Git 等系统（二次开发） |

**支持文档格式**：PDF、Word、Excel、TXT、Markdown、CSV 等（适合导出后的 Wiki / 飞书文档）。

## 技术栈

- **核心框架**：LangChain（RAG 检索增强生成）
- **后端服务**：FastAPI
- **前端界面**：Streamlit
- **向量存储**：FAISS（默认）/ Milvus / PGVector / Elasticsearch 等
- **部署方式**：内网 K8s / 虚拟机私有化部署，支持本地模型离线运行

## 快速部署（内网 PoC）

### 环境要求

- Python 3.10 ~ 3.11
- Linux 服务器（推荐）/ Windows
- 建议 8GB+ 内存（本地模型建议 16GB+）

### 安装与启动

```bash
cd libs/chatchat-server
poetry install
poetry run chatchat init
poetry run chatchat start -a
```

启动后访问 WebUI（默认 `http://127.0.0.1:8501`），即可创建第一个知识库。

### 互联网团队试点建议

```
选定高频痛点场景 → 导出文档入库 → 研发/客服小范围试用 → 优化检索与提示词 → 按业务线扩展
```

**推荐优先试点的三个场景**（见效快、文档相对规范）：

1. **客服 / 运营 FAQ 库** — 活动规则、会员权益、退款流程（缓解重复答疑）
2. **核心接口 / 微服务文档库** — 前后端联调时查字段与逻辑（减少 @ 后端同事）
3. **On-Call Runbook 库** — 常见告警与故障预案（缩短告警处理时间）

**试点步骤**：

1. 从飞书 / Confluence 导出 Markdown 或 PDF，上传至独立知识库
2. 邀请 5~10 名目标用户（研发或客服）试用 1~2 周
3. 对比「问系统」与「翻 Wiki / @ 人」的耗时与准确率
4. 根据反馈调整分块大小、检索 Top-K、提示词口径
5. 试点通过后，按业务线（交易、用户、增长等）逐步扩展

### 配置说明

| 文件 | 说明 |
|------|------|
| `basic_settings.yaml` | 服务端口、日志、跨域等 |
| `model_settings.yaml` | 内网模型地址或公司已采购的 API |
| `kb_settings.yaml` | 知识库路径、检索与分块参数 |
| `prompt_settings.yaml` | 问答提示词（可定制产品 / 客服口径） |

## 项目定位

> **智知库 = 互联网企业内部文档 + 大模型理解力 + 私有化部署 + 可溯源回答**

不是替代飞书、Confluence 或 GitLab，而是在此之上增加一层**「能对话、有出处、跟得上迭代」**的智能检索能力——让研发少 @ 人、让客服少翻文档、让 On-Call 少踩坑。
